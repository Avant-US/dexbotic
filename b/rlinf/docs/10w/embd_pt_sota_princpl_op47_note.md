## 1. Episode 与 Action Chunk

### 1.1 Episode（回合/片段）

Episode 是强化学习和机器人领域的基本单位，指**一次完整的任务执行过程**——从环境 reset 到任务结束（成功、失败或截断）。

例如：机器人从桌上抓起一个杯子放到指定位置，从 reset 开始到杯子放好（或超时），这整个过程就是 1 个 episode。一个 episode 包含若干个时间步（timestep），每步有观测、动作、奖励。

文中的典型用法：
- "Open X-Embodiment: 1M+ episodes" — 数据集包含超过 100 万次完整任务执行记录
- "单 episode 仿真 unit test" — 跑一次完整任务作为最小测试
- "失败 episode 全程记录回流到训练集" — 把失败的完整尝试录下来用于持续学习

### 1.2 Action Chunk（动作块）

Action Chunk 是模型**单次推理输出的一段连续动作序列**，而不是只输出当前一步的动作。

例如：模型一次推理输出未来 50 步（约 1 秒 @50Hz）的动作轨迹，这 50 步就是一个 action chunk。实际执行时通常只执行前 10–20 步，然后重新推理（**Receding Horizon**，滚动窗口），类似 MPC 的滚动执行。

文中的关键细节：
- "Action Chunk 长度 0.5–1 s（50 Hz 下 25–50 步）是当前甜蜜区" — 太短会闭环抖动，太长泛化差
- "推 50 步只执行前 10–20 步，然后重推" — 滚动窗口策略
- 来源于 **ACT（Action Chunking Transformer）**，最早在 ALOHA 双臂系统中提出

### 1.3 异同对比

| 维度 | Episode | Action Chunk |
|------|---------|--------------|
| **粒度** | 宏观：一次完整任务（几秒到几分钟） | 微观：一次推理的输出（0.5–1 秒） |
| **包含关系** | 1 个 episode 包含**很多个** action chunk | 多个 action chunk 串起来构成 1 个 episode |
| **语义** | "做了一次任务" | "模型想了一下，计划了接下来这段动作" |
| **数量级** | 训练数据集以 episode 计（万~百万级） | 一个 episode 内可能执行几十到上百个 chunk |
| **谁决定边界** | 环境的 done/truncation 信号 | 模型架构设计（chunk size 是超参） |

```mermaid
flowchart LR
    subgraph EP ["Episode（一次完整任务）"]
        direction LR
        C1["Action Chunk 1<br/>模型推理 50 步<br/>执行前 15 步"]
        C2["Action Chunk 2<br/>模型推理 50 步<br/>执行前 15 步"]
        C3["Action Chunk 3<br/>..."]
        CN["Action Chunk N<br/>任务完成 / 超时"]
        C1 --> C2 --> C3 --> CN
    end

    RESET["env.reset()"] --> EP
    EP --> DONE["done / truncated"]
```

简单类比：episode 相当于"一盘棋"，action chunk 相当于"想好接下来几步棋一起落子"。

## 2. 相机参数随机化（Sim2Real 关键）

### 2.1 问题背景

原文：

> **相机参数随机化**(intrinsic + extrinsic + 噪声)——很多人忘做,导致 sim2real 第一次部署直接 0%。

在仿真环境中训练的视觉策略（VLA / visuomotor policy），如果仿真中的相机设置和真实相机**完全一致**（固定参数、无噪声），模型就会**过拟合到仿真相机的完美成像条件**。一旦部署到真机，真实相机的参数哪怕和仿真只差一点点，模型就"认不出"场景了，成功率直接归零。

解决方法是**在仿真训练时随机扰动相机参数**（Domain Randomization），让模型见过各种"不完美"的成像条件，从而对真实相机的微小差异具备鲁棒性。

### 2.2 Intrinsic（内参）

**内参描述相机自身的光学和成像属性**——"镜头和传感器本身的特性"。

| 参数 | 含义 | 真机中的来源 |
|------|------|-------------|
| 焦距 $f_x, f_y$ | 镜头到传感器的等效距离（像素单位） | 镜头规格 + 装配公差 |
| 主点 $c_x, c_y$ | 光轴与像平面的交点（理想情况在图像正中心，实际会偏） | 装配偏差 |
| 畸变系数 $k_1, k_2, p_1, p_2 \ldots$ | 镜头导致的桶形/枕形畸变 | 镜头光学特性 |

用矩阵表示：

$$
K = \begin{bmatrix} f_x & 0 & c_x \\ 0 & f_y & c_y \\ 0 & 0 & 1 \end{bmatrix}
$$

**直觉**：内参决定了"3D 世界中的一个点，投影到图像上的哪个像素"。两台同型号 RealSense，内参也会因为制造公差略有不同。

### 2.3 Extrinsic（外参）

**外参描述相机在世界/机器人坐标系中的位姿**——"相机装在哪里、朝哪个方向"。

| 参数 | 含义 | 真机中的来源 |
|------|------|-------------|
| 平移 $t_x, t_y, t_z$ | 相机光心相对于机器人基座/末端的位移 | 安装位置 |
| 旋转 $R$（$3 \times 3$）或四元数 | 相机朝向相对于机器人坐标系的旋转 | 安装角度 |

用齐次变换矩阵表示：

$$
T_{\text{cam} \to \text{ref}} = \begin{bmatrix} R & t \\ 0 & 1 \end{bmatrix} \in \mathbb{R}^{4 \times 4}
$$

**直觉**：外参决定了"相机从哪个视角看世界"。真机上每次拆装相机、线缆拉扯、碰撞后，外参都会微变。

### 2.4 为什么"忘做就 0%"

```mermaid
flowchart TB
    subgraph SIM ["仿真训练（不做随机化）"]
        S_INT["内参 = 完美针孔模型<br/>fx=615.0, cx=320.0"]
        S_EXT["外参 = 完美安装<br/>相机在末端正上方 10cm<br/>俯角 30.000°"]
        S_NOISE["噪声 = 无"]
    end

    subgraph REAL ["真机部署"]
        R_INT["内参 = fx=612.3, cx=318.7<br/>有桶形畸变（制造公差）"]
        R_EXT["外参 = 偏了 2mm + 旋转 0.5°<br/>（安装误差）"]
        R_NOISE["噪声 = 热噪声 + 运动模糊<br/>+ 曝光波动"]
    end

    SIM -->|"模型从未见过不完美成像"| GAP["特征全乱<br/>成功率 → 0%"]
    REAL --> GAP
```

模型在仿真里从来没见过这些"不完美"，它学到的特征是**和完美成像参数强绑定的**。真机上图像稍有偏移或畸变，特征就全乱了。

**随机化的做法**（Domain Randomization）：

```python
# 训练时每个 episode 随机采样
fx = uniform(600, 630)                     # 内参随机
cx = uniform(310, 330)
distortion_k1 = uniform(-0.05, 0.05)      # 畸变随机
cam_pos += uniform(-0.005, 0.005)          # 外参平移随机 ±5mm
cam_rot += uniform(-1°, 1°)               # 外参旋转随机 ±1°
pixel_noise = gaussian(0, 3)              # 传感器噪声
```

这样模型被迫学到**对这些变化不敏感的视觉特征**，部署到真机时即使参数不完全匹配也能正常工作。

## 3. 相机外参对齐到 base / EEF

### 3.1 问题背景

原文：

> 相机外参对齐到 base / EEF [找不到出处,工程惯例]。

这是一个**数据规范化**步骤：把不同机器人、不同相机安装方式采集到的图像数据，统一转换到**相同的参考坐标系**下存储，让模型不需要关心"相机具体装在哪"。

### 3.2 两个常用参考坐标系

```mermaid
flowchart LR
    subgraph BASE ["base（基座坐标系）"]
        B["机器人底座<br/>固定不动"]
    end

    subgraph EEF ["EEF（末端执行器坐标系）"]
        E["夹爪 / 法兰<br/>随手臂运动"]
    end

    subgraph CAM_FIXED ["固定相机（头顶 / 桌边）"]
        CF["外参 = T_cam→base"]
    end

    subgraph CAM_WRIST ["腕部相机（手腕上）"]
        CW["外参 = T_cam→eef"]
    end

    CF --> BASE
    CW --> EEF
```

| 参考系 | 外参含义 | 典型场景 |
|--------|---------|---------|
| **base** | $T_{\text{cam} \to \text{base}}$：相机相对于机器人底座的位姿 | 第三人称相机（固定在外部） |
| **EEF** | $T_{\text{cam} \to \text{eef}}$：相机相对于末端法兰的位姿 | 腕部相机（跟着手臂一起动） |

### 3.3 为什么需要"对齐"

不同机器人的相机安装位置千差万别：

| | 机器人 A（Franka） | 机器人 B（UR5） |
|---|---|---|
| 头顶相机 | base 正上方 1.2m | base 斜上方 0.8m |
| 腕部相机 | 法兰左侧 5cm | 法兰正下方 3cm |

同一个杯子，两台机器人拍出来的图像视角完全不同。如果不做对齐，跨本体训练的模型会混乱。

**"对齐到 base / EEF"** 就是：

1. **固定相机** → 存储 $T_{\text{cam} \to \text{base}}$（相机到 base 的变换）
2. **腕部相机** → 存储 $T_{\text{cam} \to \text{eef}}$（相机到末端的变换）

有了这个变换矩阵，模型或数据预处理管线就可以：
- 把图像中的像素**反投影**到 base 坐标系下的 3D 点
- 把不同机器人的观测**统一到同一个空间参考系**
- 让"手眼关系"在不同本体之间可迁移

### 3.4 坐标变换链

对于腕部相机，有了 $T_{\text{wrist\_cam} \to \text{eef}}$ 和 $T_{\text{eef} \to \text{base}}$（来自正运动学），就能把腕部相机的观测转换到 base 坐标系：

$$
T_{\text{wrist\_cam} \to \text{base}} = T_{\text{eef} \to \text{base}} \cdot T_{\text{wrist\_cam} \to \text{eef}}
$$

```mermaid
flowchart LR
    WC["腕部相机坐标系"] -->|"T_cam→eef<br/>（外参，标定得到）"| EEF["末端执行器坐标系"]
    EEF -->|"T_eef→base<br/>（正运动学实时计算）"| BASE["基座坐标系"]
    WC -->|"T_cam→base = 两者相乘"| BASE
```

### 3.5 数据集中的实际存储

在数据集中，每条轨迹除了存图像本身，还要存外参矩阵：

```python
{
    "image_head":             jpeg_bytes,        # 头顶相机图像
    "image_wrist_left":       jpeg_bytes,        # 左腕相机图像
    "T_head_to_base":         [4x4 matrix],      # 头顶相机外参（相对 base）
    "T_wrist_left_to_eef":    [4x4 matrix],      # 腕部相机外参（相对 EEF）
    "eef_pose_in_base":       [4x4 matrix],      # EEF 在 base 下的位姿
    "actions":                [...],
    ...
}
```

**一句话总结**："相机外参对齐到 base / EEF" = 在数据集中统一记录每个相机相对于 base（固定相机）或 EEF（腕部相机）的位姿变换矩阵，是跨本体（cross-embodiment）预训练的基础工程步骤。

## 4. SOP（Standard Operating Procedure，标准操作规程）

### 4.1 含义

SOP = **Standard Operating Procedure**，即**标准操作规程/标准作业流程**。这是制造业、医疗、实验室领域的通用术语，被机器人数据采集领域直接借用。

在文中"数据采集 SOP"的语境下，它指的是**采集机器人示教数据时应遵循的一套标准化流程规范**——包括环境布置、相机标定、遥操作操作规范、数据命名/存储约定、质量检查步骤等，确保不同操作员、不同时间段采集的数据在格式和质量上保持一致，可以直接用于训练。

### 4.2 为什么数据采集需要 SOP

机器人示教数据采集不同于互联网爬虫——它需要人类操作员物理操控机器人，每次采集都涉及：

- **环境布置**：物体摆放位置、光照条件、背景
- **设备状态**：相机标定是否有效、关节零位是否准确、夹爪是否正常
- **操作规范**：遥操作速度、轨迹风格、失败时是否继续录制
- **数据存储**：文件命名、目录结构、元数据标注

没有 SOP，不同操作员采集的数据质量参差不齐，轻则增加数据清洗成本，重则引入系统性偏差（比如某个操作员习惯性地用力过猛，或总是从同一个角度抓取）。

### 4.3 一个典型的机器人数据采集 SOP 包含

| 阶段 | 内容 |
|------|------|
| **采集前** | 相机标定检查、关节零位校准、环境光照标准化、物体摆放随机化规则 |
| **采集中** | 遥操作速度限制、失败处理流程（丢弃 or 标注后保留）、每 N 条做一次质量抽检 |
| **采集后** | 数据完整性校验、格式转换、元数据填写（任务描述、成功/失败标签、操作员 ID） |
| **数据入库** | 去重、质量评分、版本管理、与已有数据集的合并规则 |

### 4.4 文中提到的"数据飞轮"与 SOP 的关系

文中提到"数据采集是否形成了标准流程 SOP 和循环的数据飞轮"——这是指一个**自我强化的闭环**：

```mermaid
flowchart LR
    SOP["标准化采集 SOP"] --> DATA["高质量数据"]
    DATA --> TRAIN["训练更好的模型"]
    TRAIN --> DEPLOY["部署到真机"]
    DEPLOY --> FAIL["失败 episode 回流"]
    FAIL --> DATA
    DEPLOY --> IMPROVE["改进 SOP<br/>（发现采集盲区）"]
    IMPROVE --> SOP
```

SOP 是飞轮的起点——没有标准化的采集流程，就无法保证数据质量，飞轮就转不起来。

## 5. 动作 vs 视觉一致性（光流和末端运动方向相关性）

### 5.1 核心直觉

文中原文：

> **动作 vs 视觉一致性**(光流和末端运动方向相关性) [找不到出处,工程经验]。

这是一个**跨模态对齐验证**：用视觉模态（光流）独立验证动作模态（末端运动标签）的正确性。其物理直觉很简单——**机械臂动了，画面就该变，且变化方向应当一致**。

想象一个遥操作采集的抓取轨迹：

```
帧 t:   机械臂末端向右平移 5cm
帧 t+1: 相机画面中，末端工具和附近物体像素整体向右偏移
```

**光流**（optical flow）正好捕捉的就是"帧间像素运动向量场"。如果末端执行器（end-effector）的动作标签说"向右移动 5cm"，同时画面中末端区域的光流向量也指向右方，那么**动作标签和视觉观测是一致的**。

### 5.2 什么情况说明数据有问题

| 异常现象 | 可能原因 | 后果 |
|---------|--------|------|
| 动作标签说"向右"，但光流显示末端没动或向左 | 动作记录延迟/错帧、遥操卡顿 | 模型学到错误的视觉-动作映射 |
| 光流大幅运动，但动作标签全为零 | 相机晃动、外部干扰、动作丢包 | 模型学到"看见运动也不动" |
| 光流方向和末端动作方向夹角 >90° | 时间戳不对齐、帧率不匹配 | 反向策略 |

### 5.3 为什么需要这个过滤器

VLA/模仿学习的训练数据通常来自人类遥操作（Spacemouse、VR 头显、GELLO 等），采集环节有很多出错机会：

```mermaid
flowchart LR
    subgraph 采集端
        VR["VR 控制器<br/>60-90 Hz"]
        CAM["相机<br/>30 Hz"]
        JOINT["关节编码器<br/>200 Hz"]
    end

    subgraph 对齐问题
        SYNC["时间戳同步<br/>可能有 10-50ms 偏差"]
        DROP["帧丢失<br/>USB 带宽竞争"]
        LAG["网络延迟<br/>VR→机器人 RTT"]
    end

    subgraph 数据包
        IMG["图像帧 t"]
        ACT["动作标签 a_t"]
    end

    VR --> SYNC
    CAM --> SYNC
    JOINT --> SYNC
    SYNC --> DROP
    DROP --> LAG
    LAG --> IMG
    LAG --> ACT

    IMG -.- ACT

    style IMG fill:#e1f5fe
    style ACT fill:#fff3e0
```

当图像帧和动作标签来自不同传感器、不同频率、经过不同处理管线时，**帧-动作错配**是常见的数据质量问题。光流提供了一种**不需要额外标注的自动验证手段**。

### 5.4 具体实现思路

一个典型的过滤管线：

```
对每条轨迹的每一步 (frame_t, frame_{t+1}, action_t):

1. 用 RAFT/FlowFormer 计算光流 F(t→t+1)

2. 在末端执行器区域（可用分割或 bounding box）
   提取平均光流向量 f_ee = mean(F[ee_mask])

3. 将 action_t 的平移分量投影到图像平面
   得到预期像素运动方向 a_proj

4. 计算方向一致性:
   cos_sim = dot(f_ee, a_proj) / (|f_ee| * |a_proj|)

5. 过滤规则:
   - cos_sim < 0 (方向相反): 标记为"严重错帧"
   - |f_ee| 很大但 |action_t| ≈ 0: 标记为"幽灵运动"
   - |f_ee| ≈ 0 但 |action_t| 很大: 标记为"动作丢失"
```

**核心公式**很简单——就是两个向量的余弦相似度：

$$\text{consistency} = \frac{\mathbf{f}_{ee} \cdot \mathbf{a}_{proj}}{|\mathbf{f}_{ee}| \cdot |\mathbf{a}_{proj}|}$$

### 5.5 相关研究支撑

虽然没有单篇论文完整定义"光流-动作方向一致性过滤"，但多条研究线索共同支撑了这个工程实践：

**(1) 光流作为动作的物理代理**

[LAOF](https://arxiv.org/html/2511.16407)（arXiv 2511.16407）明确指出：光流捕捉的是帧间像素级运动，"天然抑制背景、强调运动物体"。它将光流作为**伪监督信号**来约束潜在动作表示的学习，证明光流约束显著提升了潜在动作表示的质量——更快收敛、更小方差、更高动作准确率。这从侧面验证了**光流方向 ≈ 物理动作方向**这个假设。

**(2) Green-VLA 的光流时间对齐**

[Green-VLA](https://arxiv.org/html/2602.00919)（arXiv 2602.00919）用光流幅度做**跨数据集速度归一化**：通过腕部相机的平均光流幅度估算执行速度，确保不同数据集间"相似的视觉变化对应相似的动作增量"。这间接依赖了光流和动作的相关性假设。

**(3) 光流→末端轨迹的直接转换**

[RIGVid](https://rigvid-robot.github.io/static/pdf/rigvid.pdf) 和 AVDC 直接从生成视频的光流中**反解出 6DoF 末端轨迹**，证明光流可以被转换为末端执行器动作。但他们也发现光流累积误差会导致轨迹不合理，这反过来也说明光流和动作之间的一致性可以作为"数据是否合理"的判据。

**(4) Motion Tracks：光流作为动作表示**

[Motion Tracks](https://arxiv.org/html/2501.06994) 将动作直接表示为图像空间中的 2D 轨迹（本质上就是光流的稀疏版本），证明了**视觉运动方向 = 动作方向**可以作为跨本体的统一表示。

**(5) FOFPred：未来光流预测驱动控制**

[FOFPred](https://arxiv.org/html/2601.10781) 证明：比起只看 RGB 帧，让模型额外预测**未来的光流**能显著提升控制质量。这说明光流蕴含的运动信息确实是动作决策的关键因素。

### 5.6 局限性

| 局限 | 原因 | 应对 |
|------|------|------|
| **旋转动作光流小** | 末端原地旋转时像素位移小，光流可能近零 | 只过滤平移为主的步骤 |
| **遮挡** | 末端被物体遮挡时光流不可靠 | 使用腕部相机（如 RealSense Wrist）而非第三人称 |
| **背景运动干扰** | 其他物体移动、光照变化产生杂散光流 | 用末端执行器区域 mask 聚焦计算 |
| **精细操作** | 夹爪合拢等微小动作光流信号弱 | 结合力矩信号或夹爪状态联合判断 |
| **计算成本** | 大规模数据集上跑 RAFT 有算力需求 | 可用轻量光流模型或抽帧计算 |

### 5.7 一句话总结

"动作 vs 视觉一致性"本质上是一个跨模态对齐验证：用视觉模态（光流）独立验证动作模态（末端运动）的正确性。它的理论基础是物理世界的因果关系——机械臂动了，画面就该变，且变化方向应当一致。虽然没有单篇论文完整定义这个过滤准则，但 LAOF、Green-VLA、Motion Tracks 等工作从不同角度验证了光流与物理动作之间的强相关性，使其成为一个有充分理论支撑的工程实践。
