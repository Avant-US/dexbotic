
# 机器人 VLA 领域的论文/文章一般会涉及的几个方面

## 任务
+ 通用分类
    - open-loop, close-loop, 短程任务, 长程任务, 与环境单论交互, 与环境多轮交互, 需要推理规划, 需要拆分多个子任务, 需要人机交互, 需要多轮人机交互, 中断, 中断并转做其它任务
+ 常识与科学
+ 推理, 数学, 代码
+ 物理与空间知识
+ 方法论抽取
+ VQA
+ 空间Grounding
+ 动作Grounding
+ 预测
+ 生成
+ 世界模型相关任务

## 数据及其处理
+ 数据来源,包含什么内容及其数据量.
    - 包含了什么公开数据集,以及对这些数据集的介绍.
    - 是否有自有数据,对自有数据的介绍. 是否有写数据采集方法, 对采集方法的介绍.
    - 采集数据时的遥操方式: VR, IK, SOP, 游戏手柄, 键盘, 力控遥操, 空间鼠标或其它等等
    - 数据采集是否形成了标准流程 SOP 和循环的数据飞轮, 怎样形成的.
    - 不同模态采样频率不一致时的对齐与融合.
+ 数据多样性如何:
    - 本体多样, 场景多样, 领域多样, 任务多样(open-loop, close-loop, 短程任务, 长程任务).
    - 常识与推理数据: QA, VQA, GUIAgent, VGrounding, Math, Coding, 日常常识, 物理常识, 科学常识
    - 具身数据或相似数据: 机器人操作, 机器人导航或汽车导航, GUI操作, 自动驾驶
    - 难易程度多样, 是否有失败样本, 是否有失败后纠正的样本.
    - 是否有仿真数据, 是否有人类数据, 是否有互联网的多模态常识类数据或其它等等.
    - 指令和 thinking 或 reasoning 路径的多样性
    - 不同种类数据的配比或权重是怎样的,配比和权重是固定还是可动态变化.
+ 数据格式: 观测空间, 动作空间, 图像信息, 视频信息, 相机信息, 坐标系是否统一元信息或其它等等. 格式是否统一.
    - 这些坐标系是否对齐: 相机外参, 关节, EEF, gripper
    - 动作空间和动作表征是否一致和对齐,  Joint-space, EEF 还是 Latent action或其它等等; 取值含义是否一样, 取动作绝对值, 增量还是joint residual或其它等等; 不同本体或任务的同一动作维度的含义是否一样; Action chunk长度多少.动作的归一化与去归一化: 短窗口归一化, 跨本体动作量纲对齐, 反量化误差.
    - 机器人 Proprioception 状态的归一化,编码方式, 不同摄像头的对齐和关键帧处理或其它等等.
    - Tokenizer: 动作有没有tokenize, 怎么做的. 图像, 视频等其它模态的tokenize又是怎么做的.
    - 跨本体身份注入, 如何标识不同类型的本体: embodiment id, Soft Prompt embodiment id, Per-embodiment readout head / proprio adapter 或其它等等.
    - 采样基频是多少Hz, 不同来源, 不同种类的数据是否一致和对齐. 观测, 状态, 不同摄像头, 不同感知源, 不同模态, 动作等数据如何对齐, 在采集时如何对齐, 采集后又如何校正对齐.
    - 是否包含文本, 图片, 视频, 3D, 触觉, 红外, 雷达, 音频, 力/力矩/力反馈, 速度, 时间戳, 元信息或其它信息等等. 以及保存各类信息用的表征格式和保存格式.
+ Label 或 reward
+ 数据增广与合成
    - 数据合成, 通过插值或随机游走合成机器人行为轨迹, 通过RL模型合成机器人行为轨迹, 通过世界模型合成 或其它合成方法
    - 仿真随机化, 领域随机化, 物理参数随机化, 相机参数随机化, VLM自动调DR(域随机化)参数或其它等等
    - 感知数据增广, 图像增广, 视频增广或其它等等
    - RL outcome-driven 是突破纯监督数据上限的另一条路
    - Scaling 不只是数据量,还有 quality alignment + action unification + RL refinement
    - 多模态扰动鲁棒性
    - 极端感知条件
+ 数据质量
    - 任务, 场景和本体是否严重不相关. 比如简单地把人型和单臂等数据混在一起训会损害模型. Naive 跨本体 mixing 必然 negative transfer
    - 轨迹是否平滑, 不同模态间时间空间上是否一致已经能对应上
    - 是否去重, 是否视频图像等清晰
    - 数据过滤策略: 去重, 移除过模糊过大和异常长宽比的图像/视频, 移除QR Code等无关内容, 移除"数学密集 / 非英文 / 纯写作 / 与具身无关"等不利于具身学习的样本, 或其他过滤方案等等.
        - 如果作为世界模型,或者用于预测下一个observation, 不需要太大分辨率, 可能用latent就可以, 不需要输出完整的高清的图片, 所以此时也不必过滤模糊图像/视频, 只要是ground truth即可.
+ 数据是否开源, 数据处理的代码是否开源.

## 模型结构
+ 模型大小, 参数多少, 是否有MoE或MoT, 是否训了多个尺寸.
+ 各模态用什么做embedding和tokenizing.
+ 是否有3D / 几何先验显式建模, 点云, 4D时空理解, 3D感知与表征或其它等等.
+ 跨本体身份注入, 如何标识不同类型的本体: embodiment id, Soft Prompt embodiment id, Per-embodiment readout head / proprio adapter 或其它等等.
+ 不同模态如何融合.如果有触觉, 触觉如何融合.
+ VLM或Backbone: Qwen, Cosmos-Reason, Gemma 或其它等等.
+ 是否用了世界模型, 如何用的. 世界模型或 WAM 和 VLA 如何集成. 世界模型或 WAM 的输入输出是什么.
+ Action Head: Autoregressive token, Flow Matching, Diffusion, Latent Action或其它等等.
+ 训练任务及其Loss Function是怎样的. 是否有 affordance head. 

## 系统
+ 系统架构是怎样的. 分几层, 单层就是规划和执行都一个模型干了, 双层就是一个慢系统(System 2)负责规划一个快系统(System 1)负责执行, 三层就是除了规划和执行还多了一个对平衡的瞬时调节, 当然还可以有其它的分层.
+ 数据血缘和数据版本的管理
+ 训练-推理一致性, 各种参数和配置在训练和推理时是否对齐: chunk 长度, 模板分布, 超参, observation获取方法, 数据处理方法, 模型计算方法. 
+ 推理时是否支持显式的"模式切换"比如: 高频纯执行模式, 低频规划+高频执行模式或其它模式等等.
+ 软件硬件层面怎么部署和连接. 有边云协同吗. 用的什么硬件, 什么算力. 
+ IO优化, 比如网络延迟, 网络抖动, 怎么优化网络速度的. 其它IO速度, 比如磁盘, 缓存等, 是怎么优化的.
+ Receding Horizon 是怎么做的
+ 延迟问题怎么解决. 延迟多少, 最大的latency来自那里, 吞吐多少, 能支持多少Hz的动作输出.
+ 有没有考虑高可用, 降级, 自动纠错与重试等系统稳定性的方案.
+ 输入数据或模态部分缺失时是否鲁棒, 是否会自动切换相应模式或者降级.
+ 机器人安全约束: 力 / 力矩硬限位、EE / 几何 / 关节安全空间、E-stop watchdog, 指令安全过滤(VLM 安全分类器拦截危险指令)或其它保证机器人和工作人员安全的措施等等.
+ 对系统和模型的稳定性是如何监控的, 比如OOD 检测, 计算资源使用率, 模型效果有没有突然下降.
+ 对失败任务或系统故障是否有足够日志进行追踪, 调试, 归因, 修复与复盘. 对于失败场景是否能回流到训练数据, 加强模型效果.
+ 推理加速
    - 模型小型化: 蒸馏, 量化, 剪枝或其它小型化方法等等.
    - 推理引擎: vLLM, sgLang 或其它用于机器人领域的推理引擎.
    - 有缓存吗, 缓存怎么做的,  KV cache 共享, chunk reuse 或其它缓存方案等等
    - 模型计算优化: 重写cuda算子, 编译器优化, TensorRT, 手写算子, 算子融合, 异步计算或其它计算优化方法等等
    - 推理算法优化, 加速推理时计算的算法, 但也有可能牵涉到特殊的训练过程, 比如 Flow 步数压缩或其它等等.
+ 推理代码或框架是否开源

## Pre Training 训练流程
+ 是否分多个阶段训练,每个阶段负责什么: 通识预训练, Latent-Action 预训练, 这人第一人称数据+多本体仿真+真机数据预训练, 相似本体仿真+真机数据预训练, 落地本体仿真和真机数据微调, 强化学习(online 或 offline), 持续训练, 当然也可以有其它的多阶段划分和训练方法.
+ 超参是如何设置的: 不同的网络模块是否有不同学习率甚至冻结, 学习率的scheduler是什么; 多个Loss是否有权重; 有没有L1, L2或weight decay; Action chunk长度多少; Flow Matching / Diffusion 的训练超参; 或者其它超参等等.
+ 用什么框架, 并行方法和精度进行训练: deepspeed, zero1/2/3, megatron, 数据并行, pipeline并行, tensor并行, 混合并行, fp16, bf8, 混合精度或者其它等等.
+ 训练稳定性的手段: Async checkpoint, gradient checkpoint, 单机和分布式的训练resume, 梯度 spike 自动 rollback 或其它能监控训练是否稳定并能及时使其恢复稳定的方案等等.
+ 训练时的数据处理: 有没有数据streaming, 有没有shuffle, 有没有实时增广或改变随机seed.
+ 是否考虑到防止灾难性遗忘, 如何做的.是否考虑到训练具身智能的同时,常识和推理能力不退化,又是如何做的.
+ 训练任务和loss function是什么, 有没有考虑到open-loop, close-loop, 短程任务, 长程任务, 需要人机交互甚至多轮交互的任务, 需要调用外部工具的任务或其它等等.
+ 是否考虑到部分输入数据或模态缺失时的鲁棒性
+ 训练代码, 不同大小用途的模型checkpoint权重等是否开源

## Mid Training 中段训练
+ 有没有做mid training, 跟pre training有什么区别
+ 用了什么数据, 怎么用的. 
+ 用了什么训练方法.
+ 训练针对什么本体, 用什么任务, 什么loss function

## 微调
+ 有没有针对特定本体或任务/Loss function做微调, 具体怎么微调的, 与预训练有什么不同.
+ 用了什么数据, 怎么用的. 
+ 用了什么训练方法.

## RL强化学习
+ 预训练阶段有没有加入RL, 怎么做的, 数据怎么做, RL训练怎么做, 做了什么训练任务.
+ 后训练阶段有没有加入RL, 怎么做的, 数据怎么做, RL训练怎么做, 针对什么本体或任务做的RL.
+ Reward怎么设计, 为什么这样设计:
    - 稀疏的基于任务成败的Reward, 还是稠密的基于执行过程的每一步的评价的Reward, 或者其它的Reward类型.
    - 有用到reward model或作为critic的value model吗, 还是用基于规则或公式的reward function. 如果用了reward model或value model, 这些model又是怎么训的. Reward function的公式又是怎样的. 或者其它的Reward实现方式.
+ 在仿真跑还是在真实机器人真实环境上跑, 用一台机器人还是多台机器人, 多台机器人间怎么交互的.
+ 有针对多本体去训RL吗
+ 训练过程中有人类介入的设计吗, 怎么做的, 人类介入带来什么好处

## 评测与消融实验
+ 评测维度: in-distribution, 任务, 环境, 物体, 本体, 指令, 长程, 鲁棒性, few-shot / zero-shot 能力, 跨领域能力, 成功率, Mean Maximum Rank Violatio 或其它评测指标等等.
+ 用了什么benchmark什么数据和任务去评测, 对比结果怎么样
+ 用了什么真机和真实任务去评测, 对比结果怎么样
+ 用了什么仿真环境和任务去评测, 对比结果怎么样
+ 评测是否产生灾难性遗忘. VLM 通识能力的非动作侧定性 / 定量评测: 防止模型在加入动作监督后通识能力崩溃, 通过 VQA / OCR / Grounding / CoT reasoning 检验.
+ 在close loop 或长程任务上表现如何, 在这方面采取了什么优化.
+ 有没有做消融实验, 哪些优化手段使模型得到了提升, 为什么; 哪些优化手段没使模型得到提升, 为什么.

Hybrid Gradient: 
**动作**:量化为 255-bin 的特殊 action tokens
混合损失函数: **L_text**（交叉熵） + **L_FM (Flow Matching loss)** + **L_AR (自回归交叉熵)**
梯度隔离 或 Knowledge Insulation

**Embodied Spatial Scaffolding 的层次化辅助任务**(都通过 L_AR 监督):
1. **Subtask prediction**:预测细粒度子任务描述
2. **Goal bounding box prediction**:预测目标物体/区域的 bbox
3. **End-effector trajectory prediction**:预测主视角下未来若干帧的 EEF 2D 轨迹
4. **Discrete action prediction**:预测离散 action token

> 这里**没有**单独的 affordance head,也没有专门的 reward / value head;一切辅助监督都通过自回归文本预测来实现 —— 这是一种"text-as-everything"的简洁设计。
